Cortez Architecture Phase I Why Cortez
Why Cortez

Three categories of computer-use exist. Cortez is in none of them.

There are OS-blessed computer-use APIs, vendor-hosted agent APIs, and accessibility-tree consumers. Each works in a slice. Cortez was designed to work in the slice none of them cover.

vs.

OS-blessed computer usee.g., Apple's per-app intents

Apple's App Intents and similar OS-blessed surfaces let an assistant drive an app when the app's developer has explicitly exposed actions. The result is reliable but narrow: the assistant can do exactly what each app blessed, in exactly the apps that bothered to bless something.

Cross-app workflows fall through the cracks. Long-tail apps that never wired up intents stay opaque. The user's actual desktop is a mix of blessed and unblessed surfaces, and the unblessed parts are usually where work happens.

does cross-app work and doesn't depend on app-by-app blessing. The unit of control is the keyboard, which every Mac app already accepts.

vs.

Vendor-hosted agent APIse.g., Anthropic computer-use, OpenAI CUA

Frontier-model vendors offer agent loops that take screenshots, reason about them, and emit pixel coordinates to click. They are powerful and they are slow, expensive, and brittle: every step is a cloud round-trip, every step emits coordinates instead of semantic targets, and every step pays the latency of vendor inference.

For a task that is mostly a known shortcut composition, this is wildly over-engineered. For a task whose plan can be computed on-device, the cloud round-trip is pure tax.

emits a structured plan made of named primitives — not coordinates — and runs the cheapest valid path first. Cloud is the fallback when on-device cannot decide; it is not the primary path. A user composing an email shouldn't ship the email body through a cloud planner to figure out which field is the body.

vs.

Accessibility-tree consumerse.g., Voice Control, screen readers

Voice Control and screen readers consume the macOS accessibility tree directly. When the tree is well-populated, they are fast and reliable. When the tree is broken — Electron apps with only AXIdentifier, Adobe products with no tree at all, Figma's canvas, anything inside a remote-desktop session — they degrade or fail.

The accessibility tree is also a single oracle. When it lies, there is no second opinion.

fuses VoiceOver state with visual perception of the screen. It uses VoiceOver as the primary oracle when it's working, falls through to scene-based navigation when it isn't, and preserves both readings when they disagree. Cortez is agnostic to whether VoiceOver is reliable on a given app — the pipeline has somewhere to go either way.

None of this is a marketing position. It's the consequence of building Cortez on three concrete constraints: the camera is the input because no agent is installed on the Mac; the keyboard is the output because that's what every app accepts; and VoiceOver is the bridge because it provides structured semantics when the tree exists, and the camera fills in when it doesn't. Every architectural choice falls out from those three.
