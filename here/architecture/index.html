Cortez Architecture Phase I Why Cortez
Architecture

A confidence-propagating pipeline from camera to keystroke.

Cortez sees a Mac through an iPhone camera and drives it over Bluetooth as if it were a wireless keyboard. Every stage of the pipeline carries forward what it knows and what it doesn't, so downstream stages decide rather than guess.

Cortez is a confidence-propagating perception-to-action pipeline with VoiceOver as the primary mediation layer for structured UI understanding. Each arrow is a contract with declared input, output, explicit confidence, and provenance. No stage reaches around another. No stage discards information the next stage might need.

01
Capture
Take a 4K frame from the iPhone's wide-angle camera with focus, exposure, and white balance locked. Carry forward the device state — thermal level, focus distance, frame sharpness — alongside the image.
02
Triage
Decide if this frame is worth processing. Is the screen on? Is the image sharp? Is the phone steady? If not, suggest a fix to the user instead of burning the next stages.
03
Detect and rectify
Find the Mac screen inside the frame and warp it flat. A tracker carries the screen's corners frame-to-frame so the next stage sees a clean, stable rectangle.
04
Read text
Run OCR on the rectified screen. For every text element, record the top three candidates and their confidences — never only the winner. Different regions get different reading rules: prose vs. code vs. identifiers.
05
Stabilize
Single frames flicker; real fields don't. Watch fields across frames, promote them when they're consistent, demote them when they vanish, and surface the rate of change so downstream stages know how shaky the read is.
06
Build a semantic graph
Pair labels to fields, identify icons, group elements into a structured map of the screen. Every pairing carries a confidence score so the planner can prefer high-confidence matches and ask questions about low-confidence ones.
07
Read VoiceOver
Watch the VoiceOver caption panel and the focus ring on screen. When VoiceOver agrees with the visual scene, trust both. When they disagree, preserve both readings and let the planner decide.
08
Assemble a world model
Fuse the stabilized scene, the semantic graph, and the VoiceOver state into a single picture of what's on the Mac right now — including how confident every part of that picture is.
09
Preflight
Before acting, check that the world is in a state we know how to act on. If VoiceOver is missing, if a modal is up we don't recognize, if the scene is too unstable — refuse the step and tell the user why.
10
Plan
Pick the next action. Try the cheapest path first: a known shortcut, a known cached navigation. Fall back to on-device planning. Fall back further to cloud planning only when on-device gives up. The output is always a structured plan, never raw coordinates.
11
Execute
Send keystrokes and pointer events to the Mac over Bluetooth. The Mac sees Cortez as an ordinary wireless keyboard. Nothing is installed on the Mac. Each step has an explicit pre-condition, action, and post-condition.
12
Verify
Look at the screen again. Did the field actually accept the text? Did VoiceOver confirm the right element is focused? When both agree, proceed. When they disagree, stop and escalate.
13
Learn
Record what worked and what didn't, by app and by step. Next time the same task runs, the cheapest path is the one we already know works.
Why this matters

Three properties of the pipeline

  • Every stage is measured before it is optimized. We instrument first, then tune.
  • The voter algorithm and the calibration thresholds are picked empirically against recorded evidence — not chosen by intuition.
  • No information is discarded between stages. If OCR knows it's uncertain, the planner finds out. If VoiceOver disagrees with the scene, both readings survive.

What stays, what changes

The Bluetooth HID layer, the primitive runner, the stabilizer, the security infrastructure — all of it works and isn't being rewritten. The redesign changes the seams between stages, not the stages that were already right. Roughly 15-25 percent new code, 30-40 percent modified, 40-50 percent unchanged.
